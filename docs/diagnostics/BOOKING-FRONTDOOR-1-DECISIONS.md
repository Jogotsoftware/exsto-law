# BOOKING-FRONTDOOR-1 — standalone public booking link + attorney settings

Session 2026-07-08 · branch `booking-frontdoor-1` (off `main` @ `0d91149`) · migration **0119** (frontier was 0118) · PR only.

A Calendly-style, service-AGNOSTIC public booking link — "grab time on my calendar" — at `/book/{slug}`. A prospect resolves the firm by its public slug, sees the firm's REAL available times (rules ∩ live Google free/busy), and books with only contact info + a reason. A SECOND, parallel surface: it reuses the booking primitives but has NOTHING to do with the in-service intake path (no service, no matter, no questionnaire, no workflow). The bait rung of the funnel.

## Preflight (both passed)
1. **Frontier** = 0118 (prod ledger + tree). Claimed **0119**.
2. **Google free/busy READ is already wired** — `calendar.freebusy.query` (`adapters/googleCalendar.ts:414`, via `queryBusyBlocks` → `getGoogleAvailability`/`fetchBusyIntervals`/`getBusyIntervals`), plus the `legal.calendar.busy` MCP tool. So **WP0 was NOT needed** — we are in the "read present" world. The whole feature stands on this read.

## Decisions (non-obvious rationale)
1. **The slug is a general PUBLIC HANDLE, not booking-specific.** `public_slug` is a column on `tenant` (migration 0119), unique, url-safe — deliberately not a `booking_slug`. The same handle will later front the intake link, per-service landing pages, and the client portal, and is domain-portable (a subdomain/custom-domain maps to the same slug with no data change). Path shape today: `/book/{slug}`.
2. **Public read = a `SECURITY DEFINER` resolver, not a broad RLS policy.** `tenant` has RLS (`tenant_self_select`, 0001), so an unauthenticated page (no tenant context yet) cannot resolve a slug through a normal scoped read. `public.resolve_public_firm(slug)` is the ONLY public cross-tenant read: it runs as definer (bypasses RLS) but its body returns EXACTLY two fields — the tenant id + firm name — for an active, slug-matching firm, and nothing else (never status, timestamps, or private data). `search_path` pinned; `EXECUTE` granted to public (the data it returns IS public). Booking rules are read separately, tenant-scoped, AFTER the tenant is resolved. This is the minimal public surface.
3. **Settings storage = the EXISTING `firm.booking_rules` (Contract L), extended.** The rules primitive already carried timezone / bookable days / bookable hours / slot granularity / buffer / **min lead time** / default duration, written via `legal.booking_rules.update` and edited on `/attorney/settings`. Reused verbatim; added ONE field — `meetingLengthsMinutes` (the client-picks-length feature) — with normalization (falls back to `[defaultDurationMinutes]`, so the in-service path is unaffected). The existing settings editor gained the meeting-length chips + the "your public booking link" display. No new settings entity.
4. **Booking model = a lightweight CONTACT + `booking.create` with the contact as subject — matter-LESS.** The brief wants `booking.create` reused AND no matter. `booking.create` is matter-coupled, and touching it would risk the in-service path (WP H). Resolution: create the contact via `legal.client.create` (extended with an optional `metadata` passthrough so the lead's email/phone/reason/source ride the CRM row — additive, existing callers unaffected), then call `booking.create` with the CONTACT entity as `matter_entity_id`. This reuses the pipeline verbatim, attributes the write to the public-intake actor, and records `consultation.booked` on the contact — with zero changes to `submitBooking`/`handlers/booking.ts`. (Semantic wart: a `matter_status` attribute lands on the contact; harmless, and the alternative — a new booking entity/action — was heavier for no functional gain.)
5. **NO-SIMULATE, enforced.** The public availability path calls `getGoogleAvailability` DIRECTLY — never `getAvailability`, which falls back to `getStubAvailability`. A firm with no connected calendar (or a read error) returns `configured:false` with zero slots; the page shows an honest "hasn't connected a calendar yet." Real Google busy-time or honestly none — never fabricated slots.

## Acceptance — receipts (jfcarzprfpoztxuqykoe; writes in the sandbox tenant)
**A — slug + public read.** `resolve_public_firm('exsto-sandbox')` → `(…00fe-…0001, "Exsto Sandbox")`; `('pacheco-law')` → `(…0001, "Pacheco Law Firm")`; `('nope-unknown')` → no row. Returns ONLY `(tenant_id, firm_name)` — no private tenant data. The function is `SECURITY DEFINER` with a two-column body (shown in migration 0119).

**B — live free/busy read.** Against the only Google-connected tenant (Pacheco, READ-ONLY — no writes to tenant-zero): `getBusyIntervals` returned `source:"google"` (the real freebusy read fired against the connected calendar, not stub/disconnected), firmActor `e193d11c…`. (Pacheco's demo calendar had 0 busy blocks in the window, so a specific busy=T could not be shown without a real event; the busy→excluded behavior is proven deterministically by C.)

**C — intersection (rules ∩ busy).** Unit test `public-booking-availability.test.ts` (4/4): with no busy every rules-candidate is available; a busy interval EXCLUDES its own slot; a free slot well before it stays available; and the buffer removes the slot immediately adjacent (the brief's `13:30` is excluded by a 15-min buffer around a 14:00 event — correct, stricter than the illustrative example). Pure computation extracted to `computeAvailabilityFromBusy` (no behavior change to `getGoogleAvailability`).

**D — public booking (sandbox).** A booking through the public write path: `legal.client.create` → a `client` contact carrying `contact_email=prospect@example.com`, `contact_phone=+15551234567`, `booking_reason="Discuss a mutual NDA"`, `source=public_booking`; `booking.create` with the CONTACT as subject; both actions attributed to the **system** (public-intake) actor `…00fe-…0002` (actor_type=system); `consultation.booked` recorded on the contact (scheduled_at 2026-07-11T18:00). All rows `tenant = …00fe-…0001`. (Sandbox has no Google, so the calendar-event write is null here; that write reuses the same `createCalendarEvent` path the in-service booking uses.)

**E — no-simulate.** `getPublicAvailability('exsto-sandbox')` (no Google) → `configured:false`, `slots:0`, firmName "Exsto Sandbox" — honest none, never fabricated.

**F — no double-book.** `submitPublicBooking` re-checks live availability at confirm and rejects a slot no longer free; the Google event is the reservation the next freebusy read excludes. The "busy slot removed from availability" logic is the same C intersection (unit-proven).

**G — settings write (sandbox).** `updateFirmBookingRules` set bookable Tue/Thu 13:00–17:00, 30-min grid, 15-min buffer, meeting lengths `[15,30,45]` through `legal.booking_rules.update`; a fresh `getFirmBookingRules` read back the same values (lengths `[15,30,45]`, hours 13–17) — persisted through core and feeding the public slot computation.

**H — isolation.** `git diff origin/main` for `verticals/legal/src/api/services.ts` (submitBooking), `verticals/legal/src/handlers/booking.ts`, and `apps/legal-demo/app/book/page.tsx` (the in-service booking page) is **empty** — the in-service booking step is provably untouched.

## Notes / follow-ups (not this session)
- **Per-tenant public-intake actor.** `PUBLIC_INTAKE_ACTOR` defaults to Pacheco's seeded `…0001-…0005`. Multi-tenant `/book/{slug}` for a NON-Pacheco firm needs that firm to have a public-intake system actor (the sandbox test used sandbox's own system actor `…00fe-…0002`). Provisioning one per tenant is the clean multi-tenant follow-up before a second firm goes live.
- **Migration application.** 0119 was applied to the shared DB for the live acceptance receipts (idempotent: `ADD COLUMN IF NOT EXISTS`, `CREATE OR REPLACE FUNCTION`, seed `WHERE public_slug IS NULL`). The merge re-applies it and stamps `private.vertical_migration` (ledger not stamped from this feature branch).
- **B against sandbox.** Sandbox has no Google connection, so the live busy=T→excluded proof ran read-only against Pacheco; connecting Google to sandbox (OAuth, interactive) would let the full B/D-calendar loop run entirely in sandbox.
- **Per-weekday different hours.** The rules model is bookable days + one hours window (the brief's "Tue/Thu 13–17" fits: days [2,4], hours 13–17). Different hours per day is a later extension.
